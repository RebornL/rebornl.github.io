大型网站技术架构--读书笔记

第一章

大型互联网系统特点：

  • 高并发，大流量
  • 高可用
  • 海量数据
  • 用户分布广发，网络情况复杂
  • 安全环境恶劣
  • 需求快速变更，发布频繁
  • 渐进式发展

大型网站架构发展过程：

  • 初始阶段：

  • 紧接着，应用服务和数据服务分离

  • 加入缓存：本地缓存和分布式缓存服务器上的远程缓存

    • 使用集群方式改善网站的并发处理能力和处理海量数据问题

    • 数据库读写分离，改善数据库负载能力

    • 使用反向代理和CDN加速网站访问

    • 使用分布式文件系统和分布式数据库系统

    • 使用NoSQL和搜索引擎

    • 业务差分，拆分成不同的应用，每个应用独立部署

    • 分布式服务

第二章：大型网站架构模式

网站架构模式

  • 分层：横向切分，应用软件可以分成应用层，服务层和数据层，各层之间相互依赖和调用

    （应用层还可以继续切分为视图层和业务逻辑层，服务层可以细分成数据接口层和逻辑处理层）

  • 分割：纵向切分，将不同功能和服务分割开来，便于软件的开发和维护，便于不同模块分布式部署，提高网站的并发处理能力和功能扩展。

  • 分布式：分层和分割的一个主要目的是为了切分之后便于分布式部署，将不同模块部署在不同的服务器上，通过远程调用协同工作。分布式问题：1.必须通过网络，可能对性能造成比较严重的影响；2.服务器越多，宕机的概率也越大哦；3.数据一致性问题；4.开发管理维护比较困难。一般有如下几种分布式方案：

    • 分布式应用和服务
    • 分布式静态资源
    • 分布式数据和存储
    • 分布式计算

  • 集群：多台服务器部署相同应用构成一个集群，通过负载均衡设备共同对外提供服务，提供更好的并发特性和提高系统给的可用性。

  • 缓存：将数据存发在距离计算最近的位置以加快处理速度。用在热点数据上。

    • CDN：内容分发网络，部署在距离终端用户很近网络服务商，可以就近以最快速度返回静态资源给用户
    • 反向代理：属于网站前端架构，请求首先访问到反向代理服务器，如果有其需要的静态资源直接返回，无需转发给应用服务器
    • 本地缓存：在应用服务器本地缓存热点数据，而无需访问数据库
    • 分布式缓存：专门的机器缓存热点数据

  • 异步：通过异步降低软件的耦合性，例如使用异步消息队列：

    • 提高系统可用性
    • 加快网站响应速度
    • 消除并发访问高峰

  • 冗余：服务器的冗余备份，实现服务高可用。一台宕机，可以将服务和数据转移到其他机器上。

  • 自动化：发布过程自动化：自动化代码管理，自动化测试，自动化安全检测，自动化部署。自动化监控，报警，失效转移和恢复，自动化降级，自动化分配资源。

  • 安全

第三章：大型网站核心架构要素

软件架构的五个核心要素：

  1. 性能:
    • 浏览器端，可以通过浏览器缓存，使用页面压缩，合理布局页面，减少Cookie传输等手段改善性能
    • 使用CDN，可以让用户通过访问最短路径获取数据
    • 使用反向代理，缓存热点数据，加快请求响应速度，减轻应用服务器负载压力
    • 使用服务器本地缓存和分布式缓存，加快请求处理过程，减轻数据库负载压力
    • 异步操作将请求发送至消息队列，而当前请求直接返回响应给用户
    • 在多用户高并发请求的情况下，使用集群共同对外服务，提高系统处理能力改善性能
    • 代码层面，使用多线程，改善内存管理等手段
    • 数据库方面，索引、缓存、SQL优化等等手段
  2. 可用性：高可用主要手段是冗余，应用部署在多台服务器上同时提供访问，数据存储在多台服务器上互相备份，这样即使任意一台宕机也不会造成数据丢失。
  3. 伸缩性：通过不断向集群中增加服务器来缓解不断上升的用户并发访问压力和不断增长的数据存储需求
  4. 扩展性：网站的扩展性直接关注网站的功能需求，设计网站的扩展性架构使其能够快速响应需求变换，主要手段有事件驱动架构和分布式服务：
    • 事件驱动架构：主要采用消息队列实现，消息生产者将消息发布到消息队列中，消息消费者则从消息队列中获取消息进行处理，分离了消息产生和消费，便于增加新的消息生产者任务和消息消费者任务。
    • 分布式服务：将业务和可复用服务分离开来，通过分布式服务框架调用。新增产品可以通过可复用的服务实现自身的业务逻辑，而不会对现有产品造成影响。
  5. 安全性：保护网站不收恶意访问和攻击，保护网站重要数据不被窃取。

第四章：瞬时响应：网站的高性能架构

  1. 性能测试

    • 不同视角的性能：

      • 用户角度：直观的，就是网站响应速度的快慢
      • 开发人员：响应延迟、系统吞吐量、并发处理能力、系统稳定性
      • 运维人员：基础设施性能和资源利用率

    • 性能测试指标：

      • 响应时间：指应用执行一个操作需要的时间，包括从发出请求开始到收到最后响应数据所需要的时间

      • 并发数：指系统能够同时处理请求的数目，反映系统的负载特性

      • 吞吐量：指单位时间内系统处理的请求数量，提现系统的整体处理能力，TPS（每秒事务数）是吞吐量的一个常用量化指标，HPS（每秒HTTP请求书）， QPS（每秒查询数）

      • 性能计数器：描述服务器或操作系统性能的一些数据指标，System Load、对象与线程数、内存使用、CPU使用、磁盘鱼网络I/O等指标

    • 性能测试方法：

      • 性能测试：以系统设计初期规划的性能指标为预期目标，对系统不断施加压力，验证系统在资源可接受范围内，是否能达到性能预期。
      • 负载测试：对系统不断增加并发请求以增加系统压力，直到系统的某项或多项性能指标达到安全临界值，如某种资源已经呈饱和状态，继续对系统施加压力，系统的处理能力不但不能提高，反而会下降
      • 压力测试：超过安全负载的情况下，继续对系统施加压力，知道系统崩溃或不能在再处理人和请求，以此获得系统最大压力承受能力
      • 稳定性测试：被测试系统在特定硬件、软件、网络环境条件下，给系统加载一定业务压力，使系统运行一段比较长时间，以检测系统是否稳定。

      c点是系统的最大负载点，d点位系统崩溃点

      与性能曲线相对应的是并发户访问响应时间曲线，在日常运行去减，可以获得最好的用户响应时间，随着并发用户数的增加，响应延迟越来越大，知道系统崩溃，用户失去响应。

  2. 性能优化

    1. Web前端性能优化

      • 浏览器访问优化：减少HTTP请求次数，使用浏览器缓存，使用压缩，CSS文件放在页面最上、JS放在页面最下，减少Cookie传输
      • CDN加速
      • 反向代理：具有安全保护功能之后，还可以配置缓存加速Web请求和负载均衡的请求，转发请求到合适的应用集群上，提高系统总体处理能力进而改善网站高并发情况下的性能。

    2. 应用服务器性能优化

      • 分布式缓存：二八定律。（这个建议详细阅读书中提高的缓存问题，缓存架构等等内容）

      • 异步操作：使用消息队列将调用异步化，改善网站的扩展性；消息队列还具有削峰作用

      • 使用集群：将并发请求分发到多台服务器上处理，避免单一服务器因负载压力过大而响应缓慢，使用户请求具有更好的相应延迟特性

      • 代码优化：多线程，资源复用（单例模式和对象池），数据结构，垃圾回收

    3. 存储性能优化

      • 机械硬盘和SSD
      • B+树和LSM（N阶合并树）
      • RAID和HDFS

第五章：万无一失：网站的高可用架构

网站的可用性（Availability）描述网站有效访问的特性，还是英文更为直接就是availability。

  • 高可用的网站架构：主要手段是数据和服务的冗余备份及失效转移

  • 高可用的应用层：应用层主要处理网站应用的业务逻辑，一个显著的特点是应用的无状态性，具有幂等性

    • 通过负载均衡进行无状态服务的失效转移

    • 应用服务器集群的Session管理：利用Session实现状态记录，Session管理有以下几种方式：

      • Session复制

      • Session绑定：可以根据Hash算法，将不同请求分配到不同服务器上

      • 利用Cookie记录Session

      • Session服务器：便于统一管理和集群

    • 高可用的服务层：可复用的服务模块为业务产品提供基础公共服务，大型网站中这些服务都是独立部署的，被具体应用远程调用。可复用的服务也是无状态的服务。

      • 分级管理
      • 超时设置
      • 异步调用
      • 服务降级
      • 幂等性设计

    • 高可用的数据：

      • CAP原理:C是数据一致性，A是数据可用性，P是分区忍耐性

        • 数据持久性

        • 数据可访问性

        • 数据一致性

  • 数据备份：

    • 冷备份

      • 热备份：异步和同步两种方式

      • 失效转移

        • 失效确认
        • 访问转移
        • 数据恢复

第六章：永无止境：网站的伸缩性架构

伸缩性是指不需要改变网站的软硬件设计，仅仅通过改变部署的服务器数量就可以扩大或者缩小网站的服务处理能力。伸缩性可以分成两类：一类是根据功能进行物理分离实现伸缩；一类是单一功能通过集群实现伸缩。

  • 物理分离：新增服务器总是从现有服务器中分离部分功能和服务

    分层之后，还可以根据以下两种情况进行分离，实现系统的伸缩性

    • 纵向分离：将业务处理流程不同部分分离部署，实现系统伸缩性

    • 横向分离（业务分割后分离）：将不同的业务模块分离部署，实现系统的伸缩性

  • 单一功能通过集群规模实现伸缩

分布式缓存集群的伸缩性

分布式缓存数据各不相同，这一点与应用服务器不同，制约着缓存集群的伸缩性设计，所以需要考虑新上线的缓存服务器对整个分布式缓存集群影响最小，也就是新加入的缓存服务器应使整个缓存集群中已经缓存的数据尽可能被访问到。

  • Memcached分布式缓存集群

    路由算法负责选择缓存写入到哪个服务器或者从哪个服务器读取缓存数据。

    分布式缓存扩容，一种解决方法，在网站访问量最小的时候扩容缓存服务器集群，这个时候对数据库的负载冲击最小，然后再通过模拟请求的方法预热缓存，使缓存服务器中数据重新分布。另外一种算法，改进路由算法，减少新增缓存服务器带来的影响，如采用一致性Hash算法（详细过程请看书）

第七章：网站的可扩展架构

扩展性是指现有系统影响最小的情况下，系统功能可持续扩展或提升的能力。这是系统架构设计层面的开闭原则（对扩展开放，对修改闭合），架构设计考虑未来功能扩展，新增功能不需要对现有系统结构和代码进行改造。

构建可扩展的网站架构

核心思想是模块化，并在此基础上，降低模块之间的耦合性，提高的模块的复用性。模块化设计可以采用分层和分割分成独立的组件模块，这些组件模块通过消息队列或者依赖调用聚合成一个完成的系统。

模块分布式部署之后，聚合的主要方式有分布式消息队列和分布式服务。

分布式消息队列降低系统的耦合性

  • 事件驱动架构：（Event Driven Architecture），通过在低耦合模块之间传输事件消息，保持每个模块的松耦合，借助事件消息的通信完成模块间的合作。常见的架构设计是生产者消费者模式。通过分布式消息队列实现。

    分布式消息队列
    • 伸缩性方面：消息队列服务器上的数据可以看做是被即时处理，因此类似与无状态的服务器
    • 可用性方面：为了避免消费者进程处理缓慢，在内存不足，内存队列已满的情况下，会将消息写入到硬盘中，直至内存中消息处理完，再次从硬盘加载。
    • 保证消息被消费处理，只有消息真正被消费后才会删除消息。消息队列服务器宕机后，生产者服务器会选择分布式消息队列服务器集群中其他服务器发布消息。

分布式服务打造可复用的业务平台

分布式服务接口通过接口分解系统的耦合性，不同子系统通过相同的接口描述进行服务调用。

巨无霸应用的几个问题：

  1. 编译、部署困难
  2. 代码分支管理困难
  3. 数据库连接好近
  4. 新增业务困难

解决方案：

  • 纵向拆分：将一个大应用拆分成多个小应用
  • 横向拆分：将复用的业务拆分出来，独立部署为分布式服务

第八章：网站的安全架构

网站应用的攻击和防御

常见攻击方式

  • XSS攻击：跨站点脚本攻击（Cross Site Script），黑客通过篡改网页，注入恶意脚本，在用户浏览时，控制用户浏览器进行恶意操作。

    • 反射型攻击：诱使用户点击一个嵌入恶意脚本链接

    • 持久性攻击：黑客将恶意脚本保存到数据库中，当用户进行页面浏览时，就会自动下载执行恶意脚本

防范XSS攻击，主要有以下两种手段：

  1. 字符串过滤/消毒：例如对某些html危险字符进行转义，如将<转义成&lt，将>转义成&gt等

  2. HttpOnly：对Cookie添加HttpOnly属性，禁止JavaScript访问带有HttpOnly属性的Cookie

  • 注入攻击

    • SQL注入：攻击者在HTTP请求中注入恶意SQL命令，服务器用请求参数构造数据库SQL命令，恶意SQL被一起构造，并在数据库中执行。

      解决SQL注入攻击：

      1. 字符串过滤
      2. 预编译，参数绑定，可以有效防止SQL注入

    • OS注入：类似SQL注入，采用系统命令

  • CSRF攻击：跨站点请求伪造，Cross Site Request Forgery，黑客通过跨站请求，以合法用户的身份进行非法操作，例如转账交易、发表评论等等。核心是利用浏览器的Cookie或者服务器的Session策略，盗取用户身份。

防御方式

  • Web应用防火墙
  • 网站安全漏洞扫描

信息加密技术和密钥安全管理

信息加密技术有如下三类

  • 单项散列加密：指通过对不同输入长度的信息进行散列计算，得到固定长度的输出。即无法从固定长度信息中获得输入消息。

  • 对称散列加密：加解密使用相同的密钥，优点明显，就是加解密效率高，系统开销小；缺点就是加解密使用同一个密钥，远程通信交换密钥有安全问题。常见对称加密算法有DES算法，RC算法等等。

  • 非对称加密：加解密使用不同的密钥，对外公开用于加密的，称为公钥；另外用于解密私有的，称为私钥。非对称加密通常用在信息安全传输，数字签名等场合。

密钥安全性管理的手段主要有以下两种手段：

  1. 把密钥和算法放在一个独立的服务器上（或者专门的硬件设施上），对外提供加密和解密服务，外部应用通过调用这些服务实现信息的加解密。优点：密钥泄露概率大大降低；缺点：有可能成为应用的瓶颈，每次加解密都要进行RPC调用，系统性能开销较大。

  2. 加解密算法内置在应用系统中，密钥单独放在服务器中，为保证密钥的安全性，实际存储时会被切分存储到不同地方，兼顾安全性同时改善性能。

10年左右的技术架构

前端性能优化

网站前端是指DNS服务，CDN服务，反向代理服务，静态资源服务等应用服务器之前的部分。80%的用户请求通过前端服务返回，后端和存储端的压力骤减。

前端架构的核心是反向代理服务器Squid集群，所有请求经过LVS负载均衡分发到每台Squid服务器中，若命中则直接返回，没有才继续发送请求给Apache应用服务器集群。另外，还有CDN服务，缓存各种热点数据，用户请求直接从最近的CDN获取数据返回。

Wikipedia CDN缓存的准则：

  • 内容页面不包含动态信息
  • 每个页面有对应唯一的REST风格的URL，以便CDN快速查找并避免重复缓存
  • 在html的响应头中写入缓存控制，告知是否缓存以及缓存的有效期

Wikipedia后端性能优化

后端服务包括：缓存、存储、数据库等被应用服务器依赖的服务。这些服务大多建立在网络通信和磁盘操作基础上，是性能的瓶颈和性能优化的重灾区。

因此后端优化的主要手段是使用缓存，将热点数据缓存在分布式缓存系统的内存中，加速应用服务器的数据读操作速度，减轻存储和数据服务器的负载。

分布式存储系统Doris的高可用架构设计分析

系统架构上保证高可用的手段是冗余：服务器热备，数据多份存储

系统整体上可以分成三个部分：

  • 应用程序服务器：对存储系统发起数据操作请求
  • 数据存储服务器：负责存储数据、响应应用服务器的数据操作请求
  • 管理中心服务器：对数据存储集群进行健康心跳检测，集群扩容、故障恢复管理和对应用程序服务器提供集群地址配置信息服务等

分布式存储系统的故障类型

  • 瞬时故障：网络通信顺是中断、服务器内存垃圾回收或者后台线程繁忙停止数据访问操作响应。特点：故障时间短，在秒级甚至毫秒级系统即可自行恢复正常响应。
  • 临时故障：网卡松动导致网络通信终端，系统升级、停机维护等运维服务引起的服务关闭，硬件问题导致服务器宕机等等。特点：这些需要人工干预才能恢复正常。
  • 永久故障：硬盘损坏，数据丢失导致的永久故障。
